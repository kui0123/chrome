PAIN POINTS

AI 生成的网页，改起来比做还难

五个场景，总有一个让你崩溃过。

1

代码看不懂

AI 给了你一个网页，想微调一下样式。打开代码一看，上千行 HTML/CSS，只能改改文案，元素样式的代码根本看不懂。

"那个 button 的 class 叫 btn-primary，改了会不会影响别的按钮？"
2

描述说不准

截图给 AI，想调整某个区域，却叫不准它的名称，也描述不清想要的颜色和尺寸。

"那个稍微大点儿，这个再弄好看点儿…" — AI 烧了海量 Token 来试图理解，最终结果只能摸奖。
3

歧义酿恶果

描述有歧义，AI "觉得"它懂了，结果目标区域没变，别的地方不知道啥时候被改了。

"截图是 xxxx，用户说的是 xx，我应该按用户说的去执行。" — 学到一个新名词：原来那个位置叫"它"不叫"Ta"。
4

改得多很苦

10 个元素要调整，建个记事本一条一条写？截图圈一堆红框，最后一起反馈？累。

第 3 条需求写了一半，发现前面两条的描述又不精确了，重写。

vs AI IDE 的选元素功能

能选，但只在它自己的地盘里

Trae / Cursor / Bolt.new 等

只能在 IDE 内部、或在它生成的预览页里选元素。换一个网页、换一个域名、别人的网站？选不了。而且你得是开发者，看得懂代码。

活在浏览器里，任意网页即开即用。不挑框架、不挑域名、不挑线上还是本地，哪怕是你打不开源码的网站也能标记。

vs 截图标注工具

圈了红框，AI 还是不知道改哪儿

CleanShot X / 系统截图 / Skitch 等

截图是死图，不携带 DOM 信息。AI 收到一张图，得自己猜"这个按钮对应代码里的哪一行"，猜错了就改到别的地方去了。

直接标记真实 DOM 元素，导出带原始 HTML 和修改后 HTML 的 Diff。AI 收到的是精确指令，不是一张需要"阅读理解"的图片。

vs 浏览器设计调试工具

能调样式，但调完就没了

VisBug / CSS Peeper / DevTools 等

面向设计师和前端开发，用来"看"和"调"，但调完没有记录、没有批量管理、更不能导出给 AI 用。用完关掉，一切归零。

标记有编号、编辑有记录、一键导出结构化 Diff。不是调试工具，是"需求描述工具"，调完就是一份给 AI 的精确修改清单。

vs 自己写需求 / 记事本

写 5 条需求，3 条不精确，1 条有歧义

纯文字描述 / 记事本 / 聊天框

"那个按钮大一点""颜色好看一点"——AI 烧了海量 Token 来试图理解，结果还得靠猜。猜不准就改错，改错了又得重描述，恶性循环。

所见即所得的可视化编辑 + 精确到像素/颜色的 Diff 报告。告诉 AI："每一处都是我要改的，不要你觉得，有疑问拿出来讨论，没疑问严格执行。"
