# 项目开发规范

## 描述

本 Skill 是 HTML Diff Marker Chrome 扩展项目的**最高优先级开发规范**。任何涉及代码变更的任务都必须遵守本 Skill 定义的流程和规则。

**核心目标**：确保所有开发工作都遵循统一的标准流程，避免因忽略规范导致的操作失败、代码质量问题、版本不一致等严重后果。

## 使用场景

**本 Skill 在以下场景自动触发**：

- 用户要求开发新功能、实现特性、添加功能
- 用户要求修复 bug、修复问题、调试
- 用户要求修改代码、调整代码、重构
- 用户要求优化性能、改进代码
- 用户要求更新文档（README.md 等）
- 任何涉及对项目文件进行修改的任务

**注意**：即使任务看起来很简单，也必须完整执行本 Skill 的流程。

## 指令

### 第一步：强制读取规范文件（必须首先执行）

在进行**任何操作之前**，必须依次读取以下两个文件的**全部内容**：

1. 读取 `Project_Rule.md`（项目根目录下）
2. 读取 `extension/README.md`（扩展加载目录下）

**⚠️ 重要提醒**：
- 不读这两个文件就开始写代码，会导致操作失败、代码质量问题、版本不一致等严重后果
- 即使你"记得"文件内容，也必须重新读取 —— 因为文件可能已经更新
- 读取是第一步，没有任何例外

### 第二步：使用 TodoWrite 规划任务

完整阅读规范文件后，根据用户需求制定详细的执行计划：

1. 分析用户需求，拆解为具体的执行步骤
2. 使用 `TodoWrite` 工具创建任务清单
3. 每个任务步骤要具体、可执行、可验证
4. 确保任务覆盖：代码修改、文档同步、测试验证等完整流程

**TodoWrite 使用原则**：
- 复杂任务（3 步以上）必须使用 TodoWrite
- 多文件修改必须使用 TodoWrite
- 涉及代码+文档同步的必须使用 TodoWrite
- todo 项粒度要合理，同一文件的多个相关修改可合并为一项

### 第三步：严格遵守规范执行

在任务执行过程中，必须严格遵守 Project_Rule.md 中定义的所有规范，包括但不限于：

#### 🔴 核心操作规则（最高优先级）

| 规则 | 说明 |
|------|------|
| **Rule 0: Edit/Write 前必须先 Read** | 任何文件修改的第一步始终是 Read 目标文件 |
| **禁止"删除→重写"两步模式** | Write 可以直接覆盖，不要先 DeleteFile 再 Write |
| **禁止创建代码生成器脚本** | 直接写目标文件，不用任何中转脚本 |
| **代码与文档必须双向一致** | 改代码→同步更新 README；改 README→检查代码实现 |
| **禁止提交半成品代码** | 代码必须完整可运行 |

#### 代码风格规范

- content.js 使用 IIFE 包裹，开头加 `'use strict'`
- 防重复注入检查：`if (window.__htmlDiffMarkerLoaded) return;`
- 函数/变量命名：驼峰式 `camelCase`
- 缩进：2 空格
- 所有 CSS 类名必须以 `html-diff-marker-` 开头

#### 工具使用规范

- 修改局部内容用 Edit，全局重写用 Write
- Edit 连续失败 2 次 → 降级到 Write
- 绝对不要因为工具限制而创建生成器脚本或中转脚本

### 第四步：完成后检查清单

任务完成后，对照以下清单进行验证：

- [ ] 所有核心文件存在且非空
- [ ] JavaScript 语法检查通过（`node --check`）
- [ ] 无临时/垃圾文件
- [ ] README.md 与代码功能一致
- [ ] manifest.json 配置与实际代码行为一致
- [ ] 没有任何脚本以字符串形式包含核心源文件的完整内容

```
用户提出开发需求

    ▼

    │
    ▼

    │
    ▼
第一步：读取 README.md
    │
    ▼

    │
    ▼

    │   ├─ 遵守代码风格规范
    │   ├─ 代码与文档同步更新
    │   └─ 不创建生成器脚本
    │
    ▼
第四步：完成后检查清单
    │
    ▼

```

## 重要提醒

> **本 Skill 的要求是强制性的，不是建议。**
>
> 任何跳过本 Skill 流程直接开始写代码的行为，都可能导致：
> - 操作失败（工具链限制）
> - 代码质量问题
> - 文档与代码不一致
> - 版本混乱
> - 陷入死循环（生成器脚本陷阱、删除循环等）
>
> **严格按流程执行，反而会更快、更稳地完成任务。**
