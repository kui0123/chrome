## 四、P0 级差异详细分析

### P0-1：设置面板主题选择使用圆形色盘而非卡片网格（S-01）

**现象**: 设置面板的主题选择区域是 5 个圆形按钮横向排列，而设计稿是 2×2 卡片网格 + 自定义颜色行。

**根因**:
- JS 中 `openSettingsPanel` 函数（第 2712-2886 行）直接创建圆形 `swatch` 元素，没有构建设计稿中的卡片结构
- CSS 中只有 `.html-diff-marker-settings-swatch`（圆形）相关样式，缺少卡片网格样式
- 设计稿的主题卡片包含三部分：渐变预览条（32px 高）+ 主题名称（11px medium）+ HEX 色值（10px mono）

**涉及文件**:
- `content.js` 第 2780-2847 行（主题选择区域构建）
- `content.css` 第 3746-3771 行（swatch 样式）

---

### P0-2：设置面板自定义颜色没有输入框和应用按钮（S-02）

**现象**: 自定义颜色是一个彩色圆环（conic-gradient），点击弹出系统颜色选择器。而设计稿有独立的一行：颜色预览方块 + HEX 输入框 + 应用按钮（→）。

**根因**:
- 实现方案不同：实际代码用原生 `<input type="color">` 隐藏在彩色圆环后
- 设计稿使用文本输入 + 手动应用的交互模式

**涉及文件**:
- `content.js` 第 2818-2844 行（自定义颜色实现）
- `content.css` 第 3767-3796 行（自定义输入框样式）

---

### P0-3：编辑面板有独立的窗口控制栏（I-01）

**现象**: 编辑面板顶部色条下面有一行白色背景的"窗口控制栏"，右对齐放置折叠和关闭按钮。设计稿中没有这一行，标题直接在色条下方。

**根因**:
- JS 中 `openInspector` 函数创建了独立的 `windowControls` div（第 3213-3232 行）
- CSS 中有 `.html-diff-marker-inspector-controls` 样式（第 2046-2054 行）
- 设计稿的折叠/关闭逻辑是点击整个头部区域，而不是单独的按钮

**涉及文件**:
- `content.js` 第 3213-3243 行（窗口控制栏 + 头部）
- `content.css` 第 2046-2112 行（controls 和按钮样式）

---

### P0-4：编辑面板宽度 440px vs 设计稿 340px（I-02）

**现象**: 实际面板宽度比设计稿宽 100px。

**根因**:
- CSS 中 `.html-diff-marker-inspector` 宽度设为 `440px`
- 设计稿 `.hdm-inspector` 宽度为 `340px`

**涉及文件**:
- `content.css` 第 2006 行（width: 440px）

## 五、P1 级差异详细分析

### P1-1：设置面板开关行样式不同（S-03）

**设计稿**：
- 高度：44px（固定）
- 分隔：每个 switch-row 底部自带 `border-bottom`
- hover：无特殊背景

**实际代码**：
- 高度：由 `padding: 8px 0` 决定
- 分隔：没有 border，使用独立的 `.html-diff-marker-settings-divider` div
- hover：有背景色（`background: var(--hdm-bg-hover)`）+ 负 margin 扩展

> 注：原报告中 S-03 和 P1-7 实为同一问题的两个侧面（高度和分隔线实现方式），现合并为一项。

**根因**:
- CSS 中 `.html-diff-marker-settings-header` 使用 `justify-content: space-between`
- 设计稿 `.hdm-settings-header` 使用 `align-items: center; gap: 8px`（只左对齐）

**涉及文件**:
- `content.css` 第 3692-3700 行（settings-header）

> 注：原报告中 S-04 和 S-06 重复计数（均指向头部布局差异），现合并为一项。优先级从 P0 降为 P1（细节差异）。

#### 原因 1：类名系统不同，无法直接对照

- **设计稿**使用 `.hdm-` 前缀（共 50+ 个组件类）
- **实际代码**使用 `.html-diff-marker-` 前缀（共 80+ 个组件类）
- 两套类名完全不同，不是简单的前缀替换，结构和命名层级也有差异
- 之前的修复是"问题驱动"的（哪里坏了修哪里），而不是"系统性对照移植"

#### 原因 2：设置面板是后期补做的，没有对照设计稿

- 设计稿（v5.2-showcase）是完整的设计系统文档
- 实际代码中的设置面板是基于功能需求直接实现的，不是从设计稿移植
- 主题选择用圆形色盘是实现者的"自由发挥"，没有对照设计稿的卡片网格

#### 原因 3：编辑面板的窗口控制栏是独立设计的

- 设计稿的编辑面板没有独立的关闭/折叠按钮行
- 实际代码增加了 `inspector-controls` 这一层，是为了更明确的交互（类似桌面应用的窗口控制）
- 这是一个"自加戏"的设计决策，与设计稿不符

#### 原因 4：之前的修复只解决了"功能 bug"，没解决"样式还原度"

- 之前的 Bug 报告关注的是：功能不可用、布局错乱、显示错误等"bug"
- 本次用户反馈的是"和设计稿不一样"，是**还原度问题**，不是功能 bug
- 之前的修复清单里没有"1:1 复刻设计稿"的验收标准

#### 原因 5：缺少像素级对比验证流程

- 没有建立"设计稿 → 代码 → 截图对比"的验证机制
- 修复后只做功能验证，不做视觉还原度验证
- 导致很多细微差异（滑块大小、圆角大小、阴影有无、padding 尺寸）一直存在
