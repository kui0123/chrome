### 4.2 Bug 修复方案

#### Bug 1 & 2：编号徽章 + 拖拽把手 + 删除角标不可见（架构级修复）

**根因分析**：
- 徽章、把手、删除角标均通过 `el.appendChild()` 插入到被标记元素内部
- 使用 `position: absolute` 相对于元素定位
- 只要元素自身或其任意祖先元素设置了 `overflow: hidden`，超出边界的部分就会被裁剪
- 常见场景：卡片容器、滚动列表、标签页等父级 overflow:hidden

**修复方案：Body 层级浮层方案**

将徽章、删除角标、拖拽把手全部改为 `document.body` 的直接子节点，使用 `position: fixed` 定位，通过 `getBoundingClientRect()` 实时同步位置。

**核心机制**：

```
被标记元素（页面 DOM）          Body 浮层容器
┌──────────────────┐          ┌──────────────────────────┐
│  元素内容         │          │  ┌─徽章（fixed 定位）      │
│                  │          │  └─删除角标（fixed 定位）  │
│                  │          │  ┌─拖拽把手×8（fixed 定位）│
│                  │          │  └─...                    │
└──────────────────┘          └──────────────────────────┘
         │                              ▲
         └── getBoundingClientRect() ───┘
             实时同步位置（rAF 节流）
```

**位置同步策略（v4.3 修正：可滚动祖先链方案）**：

> **问题修正**：原方案仅在 document 上监听 scroll 事件（捕获阶段），但 scroll 事件**不冒泡**，无法捕获嵌套滚动容器（如 `overflow:scroll` 的 div）内部的滚动。v4.3 采用「可滚动祖先链」方案彻底解决此问题。

**可滚动祖先链机制**：

```
标记元素
    │
    ▼
递归向上遍历所有祖先元素
    │
    ▼
检测每个祖先的 overflow / overflow-x / overflow-y
    │
    ├─ 值为 scroll / auto / overlay → 视为可滚动祖先，绑定 scroll 事件
    │
    └─ 值为 visible / hidden / clip → 跳过
    │
    ▼
维护 scrollAncestors 数组：entryId → [ancestor1, ancestor2, ...]
    │
    ▼
所有祖先的 scroll 事件 + window scroll + resize → 统一触发 RAF 批量更新
```

**实现要点**：

1. **祖先链构建**：封装 `findScrollAncestors(el)` 函数，递归向上查找所有可滚动祖先
   - 检测属性：`overflow`、`overflow-x`、`overflow-y`
   - 判定条件：值为 `scroll` / `auto` / `overlay`（`hidden` 和 `visible` 不算）
   - 终止条件：到达 `document.documentElement` 或 `body`
   - `window` / `document` 单独处理，不计入祖先链

2. **事件绑定策略**：
   - 标记元素时：构建祖先链，为每个祖先绑定 passive scroll 监听
   - 解绑时机：删除标记 / 元素移除时，遍历祖先链逐一解绑
   - 全局监听：`window` 的 scroll + resize 事件保留

3. **性能优化**：
   - 所有 scroll 事件共享同一个 RAF 调度器，避免重复触发
   - scroll 监听使用 `{ passive: true }`
   - 祖先链缓存于 `state.scrollAncestors` Map 中，避免重复计算

4. **边界情况处理**：
   - 元素动态添加/移除 overflow 样式：暂不处理（此类场景极少，刷新即可）
   - iframe 内滚动：不在本次范围（content script 注入在顶层文档）
   - 祖先链动态变化（如元素被移动到不同容器）：标记重建时重新计算

**位置同步核心逻辑**：
- 创建统一的 `updateFloatingLayers()` 函数，遍历所有标记元素，更新其浮层位置
- 使用 `requestAnimationFrame` 进行节流，确保滚动/缩放时流畅
- 触发源：可滚动祖先的 scroll、window scroll、window resize、元素位置/大小变化
- 元素位置/大小变化时（拖拽、滑块调整）主动触发更新

**z-index 策略（v4.3 新增）**：

> **问题修正**：原方案完全未提及 z-index 规划，浮层可能被页面高 z-index 元素遮挡。v4.3 引入统一的 z-index CSS 变量体系。

**z-index 层级规划**：

```
z-index 层级（从低到高）：
  页面元素                    ← 页面自身 z-index（通常 0 ~ 数万级）
  ───────────────────────────────────────
  浮层组件层 (--hdm-z-floating)   2147483000
    ├─ 拖拽把手 (--hdm-z-handles)     +0
    ├─ 编号徽章 (--hdm-z-badge)       +10
    ├─ 删除角标 (--hdm-z-remove)      +20
  ───────────────────────────────────────
  面板层 (--hdm-z-panel)          2147483500
    ├─ 设置面板                      +0
    ├─ 编辑面板                      +10
    ├─ 组合面板                      +20
  ───────────────────────────────────────
  工具栏 (--hdm-z-toolbar)        2147483600
  ───────────────────────────────────────
  Toast 提示 (--hdm-z-toast)      2147483640
  ───────────────────────────────────────
  最高可用值：2147483647（signed 32-bit int max）
```

**实施方式**：
- 在 content.css 顶部定义 CSS 变量（挂在 `:root` 或 `.html-diff-marker-*` 上）
- 所有浮层、面板、工具栏的 z-index 统一引用 CSS 变量
- 基础值设为 `2147483000`（接近 int32 最大值，远高于绝大多数网站的 z-index）
- 通过层级递增确保内部元素顺序正确
- 如果页面有更高的 z-index（极端情况），用户可通过修改 CSS 变量调整

**CSS 变量定义示例**：
```css
:root {
  --hdm-z-base: 2147483000;
  --hdm-z-floating: var(--hdm-z-base);
  --hdm-z-handles: calc(var(--hdm-z-floating) + 0);
  --hdm-z-badge: calc(var(--hdm-z-floating) + 10);
  --hdm-z-remove: calc(var(--hdm-z-floating) + 20);
  --hdm-z-panel: calc(var(--hdm-z-base) + 500);
  --hdm-z-toolbar: calc(var(--hdm-z-base) + 600);
  --hdm-z-toast: calc(var(--hdm-z-base) + 640);
}
```

**涉及组件清单**：

| 组件 | 当前结构 | 目标结构 | 定位方式 | z-index 变量 |
|------|---------|---------|---------|-------------|
| 编号徽章 | `el > .html-diff-marker-badge` | `body > .html-diff-marker-badge-floating` | `position: fixed` | `--hdm-z-badge` |
| 删除角标 | `el > .html-diff-marker-remove-badge` | `body > .html-diff-marker-remove-badge-floating` | `position: fixed` | `--hdm-z-remove` |
| 8 方向拖拽把手 | `el > .html-diff-marker-resize-handle` | `body > .html-diff-marker-resize-handle-floating` | `position: fixed` | `--hdm-z-handles` |

**数据关联**：
- 每个浮层元素通过 `data-entry-id` 属性与标记条目关联
- 维护 `state.floatingLayers` Map 结构：`entryId → { badge, removeBtn, handles[], scrollAncestors[] }`
- 维护 `state.mutationObservers` Map 结构：`entryId → MutationObserver`（用于检测元素移除）

**事件处理变更**：
- 徽章点击、删除角标点击、拖拽把手交互：事件绑定从元素委托改为浮层元素直接绑定
- 事件冒泡不受影响（浮层在 body，不会触发元素本身的 click handler）

**组合标记适配**：
- 组合标记的组徽章（group badge）同样迁移到 body 层级
- 组合标记没有拖拽把手，无需处理

**新增/删除时的生命周期**：
- `applyMarkVisual(entry)`：
  1. 构建可滚动祖先链并绑定 scroll 事件
  2. 创建浮层元素并 append 到 body
  3. 注册到 floatingLayers
  4. 启动 MutationObserver 监听元素 DOM 移除
- `removeMark(id)` / `stripMarkerChildren(el)`：
  1. 解绑所有可滚动祖先的 scroll 事件
  2. 断开 MutationObserver
  3. 移除对应浮层元素
  4. 从 floatingLayers / mutationObservers 注销
- `loadState()` 后：遍历所有标记元素，重建浮层（含祖先链和 Observer）
- 页面 unload：自动清理

**外部 DOM 删除检测（v4.3 新增：MutationObserver）**：

> **问题修正**：原方案仅考虑了主动删除标记时的浮层清理，未考虑页面自身 JS 删除被标记元素的场景，会导致浮层残留。v4.3 引入 MutationObserver 自动检测并清理。

**实现机制**：

```
标记元素时启动 MutationObserver
    │
    ▼
监听元素的父节点（childList: true）
    │
    ▼
当观察到被标记元素从 DOM 中移除
    │
    ▼
触发清理流程：
  1. 解绑所有可滚动祖先的 scroll 事件
  2. 断开当前 MutationObserver
  3. 移除浮层元素（徽章/把手/角标）
  4. 从 floatingLayers 中移除
  5. 如该元素正在编辑，关闭编辑面板
  6. 从 state.entries 中移除该条目
  7. 调用 saveState() 持久化
```

**实现要点**：
- 封装 `setupElementRemovalObserver(entry)` 函数
- 观察目标：被标记元素的 `parentNode`
- 观察配置：`{ childList: true }`（监听子节点变化）
- 为什么观察父节点而非元素本身？因为元素被移除后，对元素自身的 MutationObserver 会自动失效
- 回调中检查元素是否仍在 DOM 中（`document.body.contains(el)`）
- 如已移除，执行清理并从 entries 中删除该条目
- 注意：父节点也可能被删除，此时需要向上追溯，但简化方案为：定期检测所有标记元素是否仍在 DOM 中（兜底机制）

**兜底机制（可选，提升健壮性）**：
- 每次调用 `updateFloatingLayers()` 时，顺带检查元素是否仍在 DOM 中
- 如不在，触发清理
- 这样即使 MutationObserver 因父节点被整体删除而失效，也能在下一次位置更新时发现并清理

**性能考量**：
- 使用单 RAF 批量更新所有浮层，而非每个元素单独 RAF
- 滚动事件使用 passive 监听器
- 元素不可见时（`display: none` 或在视口外）跳过位置计算（可选优化）
- MutationObserver 仅监听直接父节点的 childList，性能开销极小
- 每个标记元素 1 个 Observer，元素数量多时开销可接受（Observer 本身很轻量）

#### Bug 3：编辑面板无法折叠

**根因**：`makeDraggable` 函数中（content.js 第 698 行），判断点击目标是否为按钮时使用了 `t.tagName === 'BUTTON'`，但折叠按钮内部可能包含子元素（如文字节点），导致 `e.target` 不是 BUTTON 而是内部的文本节点，从而触发拖拽而非折叠。

**修复方案**：
- 将 `t.tagName === 'BUTTON'` 改为 `t.closest('button')`
- 同样适用于工具栏拖拽、组合面板拖拽等所有使用 `makeDraggable` 的场景

**涉及文件**：`content/content.js`

**存储架构采用「分层存储」策略**：

```
┌─────────────────────────────────────────────────────┐
│  存储分层架构                                         │
├──────────────────────┬──────────────────────────────┤
│  chrome.storage.local │  sessionStorage              │
│  (全局持久化)          │  (单页面会话)                 │
├──────────────────────┼──────────────────────────────┤
│ • 用户偏好设置        │ • 标记元素数据                │
│   - showBadge         │   - 位置/大小/样式/HTML 修改  │
│   - showHandles       │ • DOM 变更记录                │
│   - showShortcuts     │ • 面板位置                    │
│ • 自定义字体          │ • 组合标记数据                │
│                      │ • 当前编辑状态                │
└──────────────────────┴──────────────────────────────┘
```

**Settings 加载状态机**：

```
脚本启动
    │
    ▼
state.settingsStatus = 'loading'
state.settings = 默认值（全部 true）  ← 先用默认值渲染，避免白屏
    │
    ▼
loadSettings() 异步调用 chrome.storage.local.get
    │
    ├─ 成功
    │     │
    │     ▼
    │   合并数据到 state.settings
    │   state.settingsStatus = 'ready'
    │     │
    │     ▼
    │   触发 settingsReady 事件，更新所有依赖 settings 的 UI
    │   （徽章可见性、把手可见性、快捷键提示显示）
    │
    └─ 失败（chrome.storage 不可用/报错）
          │
          ▼
        控制台打印警告（不阻塞主流程）
        state.settingsStatus = 'error'
        state.settings 保持默认值（全部 true）
        （用户仍可正常使用，仅设置不持久化）
```

**加载期间策略**：

1. **默认值优先渲染**：脚本初始化时立即设置 `state.settings` 为默认值（全部 true），确保 UI 第一时间渲染出来，不会因等待异步加载而闪烁或空白。

2. **渐进式更新**：异步加载完成后，如果实际存储的值与默认值不同，再动态更新 UI 状态。
   - 徽章显示/隐藏：调用 `updateBadgeVisibility()` 批量更新
   - 把手显示/隐藏：如果当前有选中元素，更新其把手可见性
   - 快捷键提示：切换工具栏底部提示文字的 display

3. **加载期间用户操作的处理**：
   - 如果加载尚未完成，用户就切换了开关：
     - 先更新内存中的 `state.settings` 和 UI
     - 等待加载完成后，用用户操作的值**覆盖**加载的值（用户操作优先）
     - 然后执行 `saveSettings()` 持久化
   - 实现方式：维护 `settingsDirty = false` 标记，加载完成后检查是否为 dirty

**错误兜底**：

1. **chrome.storage 不可用**：
   - 降级为纯内存模式：设置仅在当前页面会话有效
   - 控制台打印警告：`[HTML Diff Marker] chrome.storage.local 不可用，设置将不会持久化`
   - 不影响核心功能使用

2. **存储数据损坏**：
   - 读取后校验数据结构（`typeof` 检查每个字段）
   - 字段缺失或类型错误时，使用默认值补全
   - 损坏的字段不保留，下次写入时用正确格式覆盖

3. **写入失败**：
   - `saveSettings()` 有错误回调
   - 控制台打印警告，不中断用户操作
   - 内存中保持最新值，刷新后丢失（但用户无感知）

**Settings 存储规范**：

| 项 | 说明 |
|----|------|
| **存储位置** | `chrome.storage.local` |
| **Key 名称** | `htmlDiffMarker_settings` |
| **作用域** | 全局（所有页面共享同一套用户偏好） |
| **数据结构** | `{ showBadge: boolean, showHandles: boolean, showShortcuts: boolean }` |
| **默认值** | 全部为 `true` |
| **读取时机** | 脚本初始化时异步读取，先用默认值渲染，加载完成后更新 |
| **写入时机** | 用户切换开关时立即写入 |
| **与 saveState/loadState 关系** | 完全独立，互不影响 |
| **加载状态** | `state.settingsStatus`: `'loading' | 'ready' | 'error'` |

**实现要点**：
- 封装 `loadSettings()` 和 `saveSettings()` 函数
- `loadSettings()` 返回 Promise，支持 `.then()` 链式调用
- 初始化时设置默认值 → 调用 `loadSettings()` → 加载完成后更新

```
触发源（任一）：
  ├─ window scroll
  ├─ window resize
  ├─ 可滚动祖先 scroll（元素的 scroll/auto 祖先）
  └─ 元素位置/大小变化（拖拽/滑块调整）
    │

    │

    │

    │

    │       │
    │       ├─ 不在 → 触发清理（解绑祖先事件+移除浮层+删除条目）

    │       │

    │

### 5.4 可滚动祖先链构建流程（v4.3 新增）

```

    │
    ▼

    │
    ▼

ancestors = []
    │
    ▼

    │

    │

    │     │
    │     └─ 任一值为 scroll / auto / overlay → 视为可滚动祖先
    │           │
    │           ▼

    │

### 5.6 设置加载与持久化流程（v4.3 修正：状态机）

```
脚本初始化
    │
    ▼

    │
    ▼
UI 立即渲染（使用默认值）
    │
    ▼
loadSettings() → chrome.storage.local.get('htmlDiffMarker_settings')
    │
    ├─ 成功回调
    │     │
    │     ▼
    │   校验数据结构（类型检查+字段补全）
    │     │
    │     ├─ 数据有效 → 合并到 state.settings
    │     │
    │     └─ 数据损坏 → 使用默认值补全
    │     │
    │     ▼
    │   state.settingsStatus = 'ready'
    │     │
    │     ▼
    │   settingsDirty ?
    │     │
    │     ├─ 是（加载期间用户改过开关）→ 用内存值覆盖，调用 saveSettings()
    │     │
    │     └─ 否 → 对比加载值与当前 UI 状态，不同则更新 UI
    │           （调用 updateBadgeVisibility / updateHandlesVisibility 等）
    │
    └─ 失败回调
          │
          ▼
        控制台打印警告
        state.settingsStatus = 'error'
        保持默认值，功能正常（仅不持久化）
```

### 5.7 元素外部删除检测流程（v4.3 新增：MutationObserver）

```
标记元素时 → setupElementRemovalObserver(entry)
    │
    ▼
observer = new MutationObserver(callback)
observer.observe(el.parentNode, { childList: true })
    │
    ▼
保存到 state.mutationObservers[entryId] = observer
    │
    ▼
（等待 DOM 变化）
    │
    ▼
父节点的 childList 发生变化 → 触发回调
    │
    ▼
检查 document.body.contains(el)
    │
    ├─ 元素仍在 → 不处理
    │
    └─ 元素已被移除 → 执行清理：
          │
          ├─ 1. 解绑所有可滚动祖先的 scroll 事件
          ├─ 2. observer.disconnect()
          ├─ 3. 移除浮层元素（badge/removeBtn/handles）
          ├─ 4. 从 floatingLayers 中删除
          ├─ 5. 从 mutationObservers 中删除
          ├─ 6. 从 state.entries 中删除
          ├─ 7. 如果当前选中该元素，关闭编辑面板
          └─ 8. saveState() 持久化
```

### 9.3 风险评估

| 风险项 | 等级 | 缓解措施 |
|--------|------|---------|
| 浮层同步性能问题 | 中 | 使用单 RAF 批量更新 + passive 滚动监听；元素多的场景考虑视口内可见性检测 |
| 浮层定位偏移 | 中 | 充分测试各种布局场景（fixed/absolute/sticky 父级、transform 变换等） |
| 可滚动祖先链性能开销 | 低 | 祖先链仅在标记时计算一次，缓存复用；scroll 事件已被动监听 |
| 祖先链动态变化（元素被移动） | 低 | 标记重建时重新计算；极端场景刷新即可 |
| z-index 冲突（极端高 z-index 页面） | 低 | 2147483000 级别已覆盖绝大多数场景；如遇冲突可调整 CSS 变量 |
| MutationObserver 性能开销 | 低 | 仅监听直接父节点 childList，开销极小；每个元素 1 个 Observer，轻量 |
| 父节点整体删除时 Observer 失效 | 低 | 有兜底机制：每次 updateFloatingLayers 时检查元素是否仍在 DOM |
| 设置加载闪烁 | 低 | 默认值优先渲染，加载完成后渐进更新，避免闪烁 |
| 滑块精度不足 | 低 | 保留点击数值直接输入的功能 |
| 拖拽改 closest 影响其他按钮 | 低 | 仅影响 makeDraggable 中的按钮检测，范围可控 |
| 删除同步缩放后用户反馈需求 | 低 | 如后续有需求，可基于开关组件重新添加 |
| 旧数据兼容性 | 极低 | 忽略未知字段是 JS 对象的默认行为 |
| CSS 改动引入回归 | 中 | 充分测试，重点检查边缘场景（如元素在容器边缘时的徽章显示） |
| 设置面板状态持久化异常 | 低 | settings 默认值兜底，异常时自动恢复为默认值；chrome.storage API 有回调错误处理 |
| 双击与文本编辑冲突 | 低 | 通过 isVisualEditing 状态区分，两种模式互斥 |
| chrome.storage 不可用 | 极低 | 降级为纯内存模式，不影响功能使用，仅不持久化 |

### 12.3 浮层方案技术要点速查（v4.3 更新）

| 项 | 说明 |
|----|------|
| **定位方式** | `position: fixed`，基于视口坐标 |
| **位置计算** | `element.getBoundingClientRect()` |
| **更新时机** | 可滚动祖先 scroll、window scroll、window resize、元素位置/大小变化 |
| **更新策略** | `requestAnimationFrame` 单帧批量更新 |
| **DOM 挂载点** | `document.body` 直接子节点 |
| **数据关联** | `data-entry-id` 属性 + `state.floatingLayers` Map |
| **事件绑定** | 浮层元素直接绑定，不受元素 overflow 影响 |
| **可滚动祖先链** | `findScrollAncestors(el)` 递归查找，逐一绑定 passive scroll 事件 |
| **z-index 策略** | CSS 变量统一管理，基础值 2147483000，分层递增 |
| **元素移除检测** | MutationObserver 监听父节点 childList + updateFloatingLayers 兜底检测 |

### 12.5 设置加载状态机速查

| 状态 | 触发条件 | 行为 |
|------|---------|------|
| `loading` | 脚本初始化 | 使用默认值渲染 UI，发起异步加载 |
| `ready` | 加载成功 | 合并存储值，更新 UI（如与默认值不同） |
| `error` | 加载失败 | 保持默认值，控制台警告，功能正常（仅不持久化） |

**关键变量**：
- `state.settingsStatus`: `'loading' | 'ready' | 'error'`
- `state.settings`: 当前生效的设置对象
- `state.settingsDirty`: 加载期间用户是否修改过设置（防止加载值覆盖用户操作）
